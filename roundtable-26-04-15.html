The Python Runtime

Simple framing slide. When you type python3 example.py, you're feeding a text file into a 500k-line C program. That C program IS the interpreter. Everything we walk through next - tokenizer, parser, compiler, eval loop, object system, memory management - lives inside that one binary. Your source code is just the input.

→
Tokens

→
AST
assign ┌───┴────┐ variable bin op name: x op: + ┌───┴───┐ bin op variable op: + name: y ┌─┴─┐ 40 2
→
Optimised AST
assign ┌───┴────┐ variable bin op name: x op: + ┌───┴───┐ const variable 42 name: y
→
Bytecode
LOAD_CONST 42 LOAD_NAME 'y' BINARY_ADD STORE_NAME 'x' RETURN_CONST None
Most people think Python is "interpreted." It is - but it's compiled first. Just not to machine code. It's compiled to bytecode, which is then interpreted by the VM. The key insight: by the time your code runs, it doesn't look anything like what you typed. dis.dis() is genuinely fun to play with - encourage people to try it. The five-stage pipeline mirrors the CPython internals reference: tokenize, parse to AST, optimize, compile to bytecode, cache to .pyc.

Bytecode In
LOAD_CONST 42 LOAD_NAME 'y' BINARY_ADD STORE_NAME 'x'
→
Eval Loop + Handlers (value stack)
LOAD_CONST 42 → stack: [42] LOAD_NAME 'y' → stack: [42, y] BINARY_ADD → stack: [42+y] STORE_NAME 'x' → stack: [] x = result
This is where it clicks for most people. Python isn't magic - it's a C program running a loop. The eval loop reads each bytecode instruction, the handler executes it using a value stack, and the result falls out the other end.

So... Did We Need to Know Any of That?

Joel Spolsky’s Law: all non-trivial abstractions leak. The question is when.

Discussion Starters

  • Has a leaky abstraction ever cost you? Has deep knowledge of something ever saved you?
  • What tools is it worth understanding better and what do we? LLMs, DBs, OSs, networking, cryptography...?
  • Are there some good rules of thumb for when to understand something and when to not?
  • How deep does one go - is it worth learning about the spin of the electrons in the CPU?
This is the slide that earns its keep. The entire internals tour was setup for this conversation. Don't rush the table - let people react. The car analogy lands well. The LLM question is deliberately provocative. The goal isn't to resolve the debate, it's to surface it. Leave plenty of time for discussion. Closing position: curiosity-driven depth beats mandate-driven depth. You don't need to know all of this. But the people who do tend to make better architectural decisions, debug faster, and mentor more effectively.
